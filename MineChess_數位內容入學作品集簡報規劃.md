# Mine Chess / 地雷棋：數位內容入學作品集簡報規劃

> 目標：把《地雷棋》呈現為一件完整的數位內容作品，而不是單純的技術展示。簡報重點放在「互動內容設計、遊戲規則、視覺美術、使用者體驗、AI 協作學習歷程、可延伸應用」。

## 作品定位

- 作品名稱：地雷棋 Mine Chess
- 作品類型：回合制策略戰棋、踩地雷重構、奪旗對戰、局內進化遊戲
- 完成狀態：可遊玩的完整作品，已具備 PvP、PvE、沙盒、教學、多語系、音效、音樂、AI 決策、更新日誌與後續連線對戰功能
- 申請呈現角度：數位內容企劃、互動敘事、遊戲機制設計、UI/UX、美術風格、玩家體驗、AI 輔助創作流程
- 一句話介紹：把「踩地雷」從單人解謎改造成雙人資訊戰，玩家既是掃雷者，也是佈雷者，更是奪旗的指揮官。

## 整體視覺風格

- 字體：Roboto Mono 為主，中文可搭配 Noto Sans TC 或系統黑體；標題保持等寬字體的科技感。
- 主背景：深色星空、宇宙棋盤、藍紅對立陣營，中線可加入綠色極光或流星動勢。
- 主色：
  - 深空黑：`#050816`
  - 藍方螢光：`#38BDF8`
  - 紅方螢光：`#FB7185`
  - 地雷警示黃：`#FACC15`
  - 進化紫：`#A855F7`
  - 規則白線：`#F8FAFC`
- 圖像語言：霓虹簡筆圖示、透明呼吸光、棋盤白線、脈衝爆炸、衛星式升級節點。
- 版面建議：每頁保留一個「螢光資料框」，用來承載規格、心得、規則或更新日誌；截圖頁則讓圖片成為主角，文字只做標籤與短註解。

## 10 頁簡報內容規劃

### 01. 首頁：作品與創作自述

**頁面目的**  
第一眼讓審查者知道這不是單純作業，而是一件有完整世界觀、玩法、美術與開發歷程的數位內容作品。

**建議標題**  
`Mine Chess / 地雷棋`

**副標**  
`在星空棋盤上，把踩地雷變成資訊戰與奪旗戰。`

**首頁下方螢光框內容**  
這個框可以放你最想讓老師看到的心得與作品規格摘要，建議做成「創作宣言 + 規格卡」：

- 我把經典踩地雷的「不確定性中的推理」改造成雙人對戰，讓地雷不只是障礙，而是能被放置、搬運、轉化與反制的戰術資源。
- 這件作品從最初的 AI 輔助原型，發展到完整 React / TypeScript 遊戲。過程中我學會了從「向 AI 抽卡」轉成「監督設計與程式品質」。
- 我重視的不是炫技，而是玩家能否在視覺、規則與操作回饋中感覺到：這是一場正在星空中發生的棋局。
- 規格摘要：7 x 24 棋盤、5 種職業、雙陣營奪旗、隱藏地雷、局內進化、建築系統、AI 對戰、沙盒測試、教學、多語系、LAN 連線對戰。

**畫面安排**  
大背景放遊戲主畫面或大廳截圖；標題置中偏上；下方放半透明深色螢光框。框內用左右兩欄：左邊心得，右邊規格卡。

**可放截圖**  
大廳畫面、主棋盤全景、最能代表星空與藍紅對立的畫面。

### 02. 作品核心：從踩地雷到戰棋

**頁面目的**  
用「數位內容企劃」角度說明作品的創意轉換，而不是直接進技術。

**內容重點**

- 原始靈感：踩地雷的核心是未知、推理與風險。
- 轉化方式：把單人解謎變成雙人戰棋，讓雙方同時佈雷、掃雷、奪旗與防守。
- 遊戲張力：每一步都可能是推進、試探、陷阱或心理戰。
- 作品特色：地雷具有雙向性，玩家不只躲避地雷，也能主動使用地雷。

**建議圖表**  
「經典踩地雷 → 地雷棋」轉化圖：

| 經典踩地雷 | 地雷棋轉化 |
| --- | --- |
| 隱藏地雷 | 戰場資訊戰 |
| 數字提示 | 掃雷者偵測技能 |
| 避開危險 | 佈雷、拆雷、搬雷、轉雷 |
| 單人解謎 | 雙人奪旗與心理博弈 |

**可放截圖**  
掃雷者掃描、地雷提示、爆炸或雷區畫面。

### 03. 遊戲規格與玩法流程

**頁面目的**  
用規格書的方式讓審查者快速理解遊戲可玩性與系統完整度。

**內容重點**

- 戰場大小：7 行 x 24 列。
- 主要目標：將敵方旗幟帶回己方基地。
- 回合階段：佈陣階段、思考階段、行動階段。
- 主要資源：能量、血量、地雷數量、進化條件。
- 失敗風險：踩雷、被攻擊、旗幟掉落、能量不足、時間壓力。

**建議規格圖**

| 模組 | 設計內容 |
| --- | --- |
| 地圖 | 7 x 24 格狀棋盤，中央為爭奪焦點 |
| 勝利 | 奪取敵旗並帶回基地 |
| 單位 | 將軍、掃雷兵、遊俠、製雷兵、拆彈兵 |
| 地雷 | 普通、減速、煙霧、連鎖、終極雷 |
| 節奏 | 佈陣、思考、行動，搭配倒數計時 |
| 模式 | 玩家對戰、人機對戰、沙盒、教學、連線對戰 |

**可放截圖**  
完整棋盤截圖，加上簡短標線：基地、旗幟、中線、角色、地雷、操作面板。

### 04. 角色與進化系統

**頁面目的**  
展現遊戲不是只有「棋子移動」，而是有角色定位、成長選擇與戰術分化。

**內容重點**

- 將軍：攻擊與奪旗核心。
- 掃雷兵：資訊偵測與安全路線判斷。
- 遊俠：高機動、搬運地雷、擾亂戰線。
- 製雷兵：陷阱與區域控制。
- 拆彈兵：拆雷、防禦與承受爆炸。
- 每個角色具有兩條進化路線，形成不同戰術風格。

**建議呈現方式**  
使用五張小角色卡，卡片包含：角色名稱、定位、代表色、核心技能、進化方向。

**可放截圖**  
進化樹 UI、角色資訊面板、升級後衛星式特效。

### 05. 視覺美術：星空、霓虹與對立陣營

**頁面目的**  
這頁應該很「數位內容」。讓老師看到你對風格、符號與玩家感受有明確意識。

**內容重點**

- 藍紅對立：代表雙方陣營與戰場拉鋸。
- 白色棋盤線：象徵規則、公平與棋局秩序。
- 星空背景：把棋局放大成星際戰爭，提升作品敘事尺度。
- 霓虹簡筆圖示：讓角色與地雷在深色背景中清楚辨識。
- 呼吸特效：用透明度變化表現隱藏危險與資訊不完整。
- 脈衝爆炸：用大小與顏色讓玩家直覺理解傷害與威力。

**建議視覺規格圖**

| 視覺元素 | 功能 |
| --- | --- |
| 星空背景 | 建立宇宙對弈的沉浸感 |
| 藍紅漸層 | 表示陣營對抗與中央戰線 |
| 白色邊框 | 維持棋盤可讀性 |
| 霓虹角色 | 強化職業辨識 |
| 呼吸光 | 表現潛在危險 |
| 流星與極光 | 補足背景動態，但不干擾棋盤 |

**可放截圖**  
背景、棋盤、爆炸、地雷光環、角色圖示近照。

### 06. UI/UX：讓複雜規則變得可操作

**頁面目的**  
說明你如何把高資訊量遊戲整理成玩家能理解的操作介面。

**內容重點**

- 主畫面採「上棋盤、下操作」分離，降低視線跳動。
- 左側能量資訊：讓玩家判斷本回合與下回合資源。
- 中央小隊管理：支援點選、交換、血量監控與快捷鍵。
- 右側行動操作：移動、攻擊、技能、結束回合集中處理。
- 右側日誌：記錄戰場事件、進化、系統提示，可縮放以避免資訊壓迫。
- 教學與沙盒：讓新手能理解規則，讓創作者能快速測試平衡。

**建議圖表**  
UI 區塊拆解圖：棋盤、狀態列、角色列、操作列、日誌、沙盒工具。

**可放截圖**  
操作面板、日誌縮放、教學頁、沙盒面板。

### 07. 世界觀與敘事包裝

**頁面目的**  
讓作品更像「可被發行的內容 IP」，而不是只有機制的遊戲。

**內容重點**

- 故事背景：藍色星球與紅色星球的百年協議。
- 核心幽默：「在空中哪有地雷之說？那只是太空垃圾罷了。」
- 衝突設定：殖民將軍以插旗決定星球歸屬。
- 灰色地帶：為了避開太空武器禁止令，地雷被稱為太空垃圾。
- 人物關係：兩位將軍可能是固執的親兄弟，戰爭像一場漫長的兄弟鬩牆。
- 內容延伸：可發展成角色介紹、劇情模式、宣傳短片、世界觀網站或社群貼文。

**可放素材**

- 世界觀短文節錄。
- 藍紅星球對峙示意圖。
- 角色陣營關係圖。

### 08. 開發歷程與學習心得

**頁面目的**  
把你的 AI 協作經驗轉成「學習能力與創作方法」的證據。這頁很適合申請學校，因為它展示你如何反思、調整與成長。

**內容重點**

- 第一階段：Google AI Studio 原型，快速看到可能性，也遇到閃退、介面變形與進化系統失控。
- 第二階段：進入 IDE 與 Kiro / Claude，理解 React、TypeScript、Vite 的結構，開始看得懂程式如何被拆分。
- 第三階段：Google Antigravity 與 Gemini 3 Pro，強化 UI 一致性與多模態回饋，也學會防止 AI 過度自信與誤改。
- 工程反思：從單一巨大 `App.tsx` 拆分成多個模組，理解權責分離。
- 心態轉變：從把 AI 當抽卡機，轉向把自己放在監督者與設計者的位置。

**建議引用文字**

> 我不願再把工具當作抽卡機，我要讓代碼的好壞都能歸咎於自身的責任。

**可放圖表**

學習歷程時間軸：AI Studio 原型 → IDE 透明除錯 → 模組化重構 → 上架 → AI 對戰強化 → 連線對戰。

### 09. 完成後追加功能：連線對戰與 AI 強化

**頁面目的**  
回應你提到「最後更新日誌後其實還做了很多更新」。這頁專門放後續成長，讓審查者知道作品不是寫完就停，而是持續被打磨。

**內容重點**

- LAN / PeerJS 連線對戰：支援兩台電腦透過房間或 Peer ID 進行對戰。
- 通訊協定：將玩家操作封裝成 ActionPacket，傳輸移動、插旗、觸雷、進化選擇等事件。
- 同步思路：以事件驅動同步為主，減少傳輸整個棋盤狀態的負擔。
- AI 對戰強化：從隨機單步行動，進化到威脅地圖、意圖系統、角色分工、終局策略、開局庫、假動作與玩家風格適應。
- 可觀測性：AI DevTools 顯示決策、分數、淘汰原因與調參檔，讓平衡調整更有依據。

**建議規格圖**

| 更新項目 | 對作品的意義 |
| --- | --- |
| 連線對戰 | 從單機作品走向真實玩家互動 |
| Peer ID / Room ID | 降低連線門檻 |
| ActionPacket | 讓遊戲行為可同步、可追蹤 |
| AI Threat Map | 讓人機對戰更像真人 |
| AI DevTools | 讓設計調整可以被觀察與驗證 |

**可放截圖**

連線大廳、房間 ID、雙人連線畫面、AI DevTools、難度選擇。

7. 操作面板：呈現 UI/UX。

9. 教學頁：呈現新手引導。
10. 沙盒工具：呈現設計測試能力。

## 審圖重點提醒

- 不要讓簡報看起來像程式碼成果報告。技術可以出現，但要服務「互動內容設計」。
- 每頁最多一個主要訊息，讓老師看完能記得：這頁在講規則、角色、美術、UI、故事、學習或延伸。
- 心得不要藏在最後。首頁先放一段，開發歷程再放完整版本，這樣作品一開始就有人味。
- 規格表要漂亮地做成「遊戲設定資料卡」，不要像工程文件截圖。
- 截圖頁不要塞太多字，讓畫面說話；文字只做關鍵標籤。
